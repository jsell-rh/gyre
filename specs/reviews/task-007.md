## Findings

### R1 (resolved)

- [-] [process-revision-complete] **F1 — `ConstraintViolation` event payload missing spec-required `attestation_id` field**
  `crates/gyre-server/src/constraint_check.rs` lines 720–729: The `emit_constraint_violations` function builds the ConstraintViolation event payload with fields `constraint_name`, `expression`, `action`, `agent_id`, `repo_id`, `task_id`, `timestamp`, `message`. The spec §7.5 defines the `ConstraintViolation` message kind with `attestation_id: string` as a required field — it identifies which specific attestation chain was being verified when the constraint failed. The implementation omits `attestation_id` and substitutes `task_id` (not in the spec schema). Even in audit-only mode, `attestation_id` is needed for forensic tracing: if a task has multiple attestation nodes, `task_id` alone does not identify which node failed. The spec also requires `context_snapshot: object` (the CEL context at evaluation time), which is omitted. **Fix:** Pass the attestation ID through to `emit_constraint_violations` and include it in the payload. Add `context_snapshot` with the serialized CEL context (or a representative subset).

- [-] [process-revision-complete] **F2 — `ConstraintViolation` notification priority is 3 but spec requires 2**
  `crates/gyre-common/src/notification.rs` line 110: `NotificationType::ConstraintViolation` returns priority 3. The spec §7.5 states: "Human operator's inbox receives notification (priority 2 — high but not critical, since push/merge already rejected)." The task file (task-007.md line 33) erroneously transcribed this as "priority-3" and the implementation followed the task description rather than the spec. The spec is the source of truth. **Fix:** Change `Self::ConstraintViolation => 3` to `Self::ConstraintViolation => 2` in `notification.rs`.

- [-] [process-revision-complete] **F3 — Push-time constraint evaluation hardcodes `default_branch` to `"main"`**
  `crates/gyre-server/src/constraint_check.rs` line 145: The `evaluate_push_constraints` function sets `default_branch: "main".to_string()` in the `TargetContext`. The repository's actual `default_branch` field is available in the calling context (`git_http.rs` line 400: `let default_branch_clone = default_branch;`) but is not passed to `evaluate_push_constraints`. Any CEL constraint referencing `target.default_branch` (e.g., `target.branch == target.default_branch` to check if the push targets the default branch) will silently return wrong results for repos with a non-"main" default branch (e.g., "master", "develop"). **Fix:** Add a `default_branch: &str` parameter to `evaluate_push_constraints` and pass it from `git_http.rs` where it is already resolved from the repository record.

### R2

- [-] [process-revision-complete] **F4 — Merge-time constraint evaluation hardcodes `default_branch` to `target_branch` instead of using the repo's actual default branch**
  `crates/gyre-server/src/constraint_check.rs` line 365: The `evaluate_merge_constraints` function sets `default_branch: target_branch.to_string()` in the `TargetContext`. This is the same class of bug as R1 F3 (which was fixed for the push path) but persists in the merge path. The `target_branch` is the MR's target branch — often the default branch, but not necessarily (MRs can target feature branches, release branches, etc.). The repo's actual `default_branch` is available in the calling context: `merge_processor.rs` line 527 shows the `repo` variable is in scope, and `Repository` has `repo.default_branch` (defined at `gyre-domain/src/repository.rs:37`). A CEL constraint referencing `target.default_branch` would get an incorrect value when the MR targets a non-default branch, and the constraint `target.branch == target.default_branch` would always return `true` (since both are set to `target_branch`), making it meaningless. **Fix:** Add a `default_branch: &str` parameter to `evaluate_merge_constraints`, and pass `&repo.default_branch` from `merge_processor.rs`.

- [-] [process-revision-complete] **F5 — Dry-run button performs superficial client-side string check, not constraint evaluation against repo state**
  `web/src/lib/ConstraintEditor.svelte` lines 65–93: The spec §7.6 states: "A dry-run button evaluates the constraint set against the current repo state to preview what would pass/fail." The task acceptance criteria (task-007.md line 49) states: "Dry-run evaluation works from approval dialog." The implementation does neither — it performs client-side string heuristics: checking if expressions contain `.`, `==`, or `>=` (line 77), and if allowed-path globs contain `*` or `/` (line 83). It never calls a server endpoint, never compiles CEL expressions, and never evaluates against the actual repository state. Valid CEL like `output.changed_files.size() < 50` would be flagged as "may not be valid CEL" (it lacks `.` at a position the heuristic checks), while the syntactically invalid expression `foo == bar == baz` would pass. This is a superficial stub, not a functional dry-run. **Fix:** Add a server-side endpoint (or extend an existing one) that accepts constraint expressions and evaluates them server-side against the repo's current state using the CEL evaluator in `gyre-domain::constraint_evaluator`. Call this endpoint from the dry-run handler. At minimum, the constraints must be compiled by the real CEL parser to validate syntax — string heuristics are not equivalent.

### R3

- [-] [process-revision-complete] **F6 — ConstraintEditor strategy-implied constraints display is incomplete: missing persona, workspace trust level, and attestation level policy constraints (spec §7.6)**
  `web/src/lib/ConstraintEditor.svelte` lines 29–39: The spec §7.6 states: "The approval dialog shows strategy-implied constraints (derived from workspace config, trust level, stack policy) as read-only." The `strategyConstraints` derived block hardcodes a single `meta-spec set match` constraint and derives scope previews from the user's own input fields (`allowedPaths`, `forbiddenPaths`). It does not display: (a) persona constraints from `InputContent.persona_constraints` — the spec §3.2 derives `input.persona_constraints.exists(p, p.name == agent.persona)` from persona requirements, which are assembled server-side from the spec/workspace config; (b) workspace trust level constraints — the server's `derive_strategy_constraints` (`constraint_evaluator.rs` line 298) generates `agent.attestation_level >= 2` for supervised workspaces, but the ConstraintEditor doesn't receive a `workspace_id` prop (lines 11–17) and makes no API call to fetch workspace config; (c) attestation level policy constraints from the repo's stack policy. The server-side `derive_strategy_constraints` correctly derives all five constraint sources (persona, meta-spec, scope, trust level, attestation policy), but the UI shows a hardcoded subset. A user approving a spec in a supervised workspace will not see the attestation level constraint until push time, when their agent's code may fail constraint evaluation unexpectedly. **Fix:** Add a server-side endpoint that accepts workspace/repo context and returns the full set of strategy-implied constraints (calling `derive_strategy_constraints` with the workspace trust level and attestation level policy). Have the ConstraintEditor fetch and display them as read-only. Pass `workspace_id` as a prop from `DetailPanel.svelte`.

- [-] [process-revision-complete] **F7 — No integration tests for merge-time constraint evaluation (task acceptance criteria)**
  `crates/gyre-server/src/constraint_check.rs` lines 862–1176: The task acceptance criteria (task-007.md line 50) states: "Integration tests for push-time and merge-time evaluation." The test module contains push-time integration tests (`evaluate_push_no_attestation_chain_is_noop`, `evaluate_push_with_attestation_chain_runs_constraints`) and unit tests for helpers (`emit_constraint_violations_creates_events`, `create_violation_notifications_for_workspace_members`), but NO tests for `evaluate_merge_constraints`. The merge-time path (lines 220–431) has distinct logic: it resolves the agent from the MR, resolves the task_id from the agent, and computes the diff from a merge commit (via `compute_commit_diff` instead of `compute_push_diff`). None of this merge-specific logic is tested. **Fix:** Add at least two integration tests for `evaluate_merge_constraints`: (a) one that verifies the no-attestation-chain graceful degradation path, and (b) one that stores an attestation chain and verifies the function correctly looks up the MR, resolves the agent and task, and attempts constraint evaluation.

### R4

- [-] [process-revision-complete] **F8 — `build_agent_context` always sets `meta_spec_set_sha` to empty string, causing the unconditionally-generated meta-spec constraint to always fail and produce false violations for every constraint evaluation**
  `crates/gyre-server/src/constraint_check.rs` line 460: `build_agent_context` sets `meta_spec_set_sha: String::new()`. The `derive_strategy_constraints` function (`crates/gyre-domain/src/constraint_evaluator.rs` lines 288–291) unconditionally generates the constraint `agent.meta_spec_set_sha == input.meta_spec_set_sha`. Since the agent's `meta_spec_set_sha` is always `""` while `InputContent.meta_spec_set_sha` carries the real SHA from approval time, this constraint always evaluates to `"" == "def456"` → `false`. Under §3.4's fail-closed sequential evaluation, this false failure stops all further constraint evaluation — scope constraints, trust-level constraints, and attestation-level constraints are never reached. Every push and merge with an attestation chain produces: (a) spurious `ConstraintViolation` events (§7.5) broadcast to the workspace and directed to the agent, and (b) priority-2 inbox notifications for all workspace Admin/Developer/Owner members. The same class of issue affects `attestation_level: 0` (line 459) — in supervised workspaces, the trust-level constraint `agent.attestation_level >= 2` always fails. The `persona` field IS correctly populated (via KV store lookup, lines 447–453), demonstrating that the implementer understood the need to wire these lookups — but `meta_spec_set_sha` and `attestation_level` were left as defaults. The unit tests in `constraint_evaluator.rs` pass because `sample_agent_context()` uses correct values (`meta_spec_set_sha: "def456"`, `attestation_level: 3`), but the integration tests never reach evaluation (they use `/nonexistent/path`, causing diff computation to fail first), so this bug is untested. In Phase 2 audit-only mode, false violations create noise that drowns out real violations and makes the audit trail unreliable, defeating the purpose of establishing a baseline before Phase 3 enforcement. **Fix:** Populate `meta_spec_set_sha` from available data. The workspace's meta-spec set SHA is available via `state.workspaces.find_by_id(workspace_id)` (the workspace is already looked up on lines 95–100 for trust level). Cache the agent's meta-spec set SHA in the KV store at spawn time (same pattern as persona caching on line 449), or derive it from the workspace record. For `attestation_level`, populate from the agent's workload attestation record (`state.stack_attestations` or equivalent), or from the agent's container audit record. If the data sources are genuinely unavailable in Phase 2, guard the constraint generation: do not generate `meta_spec_set_sha` or `attestation_level` constraints when the agent context field is empty/default — similar to how `persona_constraints` is guarded by `if !content.persona_constraints.is_empty()` (line 275).
