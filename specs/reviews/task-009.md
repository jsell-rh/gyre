## Findings

### R1

- [-] [process-revision-complete] **F1 — DerivedInput signature not cryptographically verified during chain verification (spec §4.4 step 1)**
  `crates/gyre-server/src/git_http.rs` lines 1801–1817 (`verify_attestation_audit_only`, `Derived` branch): When verifying a `DerivedInput`, the function only checks that `parent_ref` is non-empty (`has_parent`). It does NOT verify the Ed25519 signature of the DerivedInput against its `key_binding.public_key`, and it does NOT check the DerivedInput's key binding expiry. Compare with the `Signed` branch (lines 1733–1798) which performs full Ed25519 signature verification (`ring::signature::UnparsedPublicKey::verify`) and key binding expiry checks. The spec §4.4 step 1 explicitly requires for DerivedInputs: `verify_signature(attestation.input.signature, attestation.input.key_binding)` and `verify_key_binding(attestation.input.key_binding)`. The `verify_chain` function (TASK-009, line 1871) calls `verify_attestation_audit_only` for each node, inheriting this gap. This means any DerivedInput with arbitrary signature bytes (e.g., `vec![0xDE; 64]`) will pass chain verification — the chain reports `valid: true` for forged delegations. **Impact:** The security property "a compromised agent cannot forge delegations" is not enforced. An attacker who can create an attestation record in the database with a DerivedInput containing arbitrary signature bytes will pass chain verification. **Fix:** Add to the `Derived` branch of `verify_attestation_audit_only`: (a) Ed25519 signature verification — compute `SHA256(serde_json::to_vec(&derivation_content))`, create `UnparsedPublicKey::new(&ED25519, &derived.key_binding.public_key)`, call `.verify(content_hash, &derived.signature)`, (b) key binding expiry check — `!derived.key_binding.is_expired(now)`. Mirror the structure of the `Signed` branch.

- [-] [process-revision-complete] **F2 — AIBOM includes chain summary instead of full attestation chain per commit (spec §7.3, supply-chain.md §5)**
  `crates/gyre-server/src/api/aibom.rs` lines 148–158 (in the TASK-009 diff): The code includes a summary object for each commit's chain attestation containing `attestation_id`, `chain_depth`, `has_signed_input`, `constraint_count`, `gate_attestation_count`, `chain_node_count`. The code comment (line 148) says "Include chain summary (not the full chain to keep response size manageable)." The spec §7.3 states: "The AIBOM (§5 of supply-chain.md) is extended to include the full attestation chain per commit, replacing the flat stack_attestation field." The spec §5.3 states: "AIBOM — release-time aggregation includes the full chain for each commit." The acceptance criterion says "AIBOM includes full attestation chains." The summary is useful metadata but does not satisfy the spec requirement for the full chain. An offline verifier consuming the AIBOM cannot reconstruct or verify the attestation chain from summary counts alone. **Fix:** Include the serialized attestation chain (the `chain` Vec loaded from `state.chain_attestations.load_chain(...)`) as the `chain_attestation` field value instead of the summary object. If response size is a concern, add an optional query parameter (e.g., `?chain_detail=summary`) to allow summary mode, but default to full chain to match the spec.

- [-] [process-revision-complete] **F3 — ProvenanceChain.svelte component not wired into the Explorer UI (spec §7.6)**
  `web/src/components/ProvenanceChain.svelte` (261 lines): The component implements the spec §7.6 requirement: "Clicking a commit shows its attestation chain as a directed graph. Each node shows signer identity, constraint count, verification status. Failed constraints highlighted." However, the component is never imported by any other file. A `grep -r "ProvenanceChain" web/src/` returns only the component's own self-reference in its doc comment. The component is dead code. The acceptance criterion says "Explorer shows provenance chain visualization for commits." The `/api/v1/repos/:id/attestations/:commit_sha/chain` endpoint is registered and functional (TASK-009 added it to `api/mod.rs`), but the UI component that consumes it is disconnected. **Fix:** Import `ProvenanceChain` into the commit detail view — likely in `web/src/lib/DetailPanel.svelte` within the commit section (around line 2943 where commit details are rendered), or as a tab in the commit view. Wire the `repoId`, `commitSha`, and `token` props from the current context. The existing commit detail view has a provenance chain (spec→task→agent→MR) but that's referential metadata, not the cryptographic attestation chain visualization that §7.6 describes.

- [-] [process-revision-complete] **F4 — Stale "audit-only" messages in verification results propagated to enforcement and API responses**
  `crates/gyre-server/src/git_http.rs` lines 1833–1837 (`verify_attestation_audit_only`): The function produces result messages saying `"all checks passed (audit-only)"` (valid case, line 1834) and `"one or more checks failed (audit-only, not rejecting)"` (invalid case, line 1836). Phase 3 enforcement (TASK-008) is active — the system now rejects pushes and merges based on these verification results. These messages appear in: (a) the verification endpoint response (`GET .../verification`) as per-node children of the `verify_chain` result, (b) the chain visualization endpoint response (`GET .../chain`) as `node.message` fields in the `ChainNode` structs (provenance.rs line 420: `message: node_result.message`). An API consumer or Explorer user sees `"audit-only, not rejecting"` for invalid nodes in a system that IS rejecting. The constraint_check.rs `log_constraint_results` function was correctly updated by TASK-009 to remove "audit-only" from log messages (diff lines 843, 858), but the `verify_attestation_audit_only` function's result messages were not updated. **Fix:** Update the result messages in `verify_attestation_audit_only` (lines 1834 and 1836) to remove "audit-only" and "not rejecting." For example: `"all structural checks passed"` and `"one or more structural checks failed"`. The function name `verify_attestation_audit_only` can remain (it describes the function's verification scope, not the enforcement mode), but the user-facing messages in the VerificationResult must not claim the system won't reject.
  **Resolution (commit e464e0d8):** Messages updated to `"all structural checks passed"` and `"one or more structural checks failed"` at git_http.rs lines 1896 and 1898. Log messages in git_http.rs lines 503 and 512 also updated to remove "audit-only" and "not rejecting" language. `scripts/check-stale-enforcement-mode.sh` passes.
