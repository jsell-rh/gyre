# Review: TASK-053 — Explorer Phase 4: Large Graph Performance & Responsive Layout

## R1 — needs-revision, 5 findings

### Findings

- [ ] **F1: Mirrored-logic tests — `shouldRenderBadges()` tests the wrong condition (zoom-level vs. actual screen-space size check).** The test defines `shouldRenderBadges(zoom) { return zoom >= 1.0; }` and asserts against it. But `ExplorerCanvas.svelte:3308-3309` controls badge rendering with `if (sw > 40 && sh > 25)` — a screen-space size threshold, not a zoom-level check. The test's logic diverges from the component's actual behavior. The test always passes (it validates its own function definition) and provides zero coverage of the real badge rendering rule. If badge rendering were broken, this test would not detect it. **Fix:** Replace the local `shouldRenderBadges()` with a test that renders ExplorerCanvas at different zoom levels and asserts on whether badge-related draw calls occur — or at minimum, rewrite the local function to match the actual component condition (`sw > 40 && sh > 25`).

- [ ] **F2: Mirrored-logic tests — `shouldRenderText()` and the `isVisibleAtZoom()` tests exercise locally-defined functions, not the component.** Three test groups (`LOD text rendering`, `large graph viewport culling logic`, `edge culling logic`) each define a local function that mirrors (with varying accuracy) a function inside ExplorerCanvas.svelte, then test the local copy. This is the "mirrored-logic test" anti-pattern: both sides of the assertion are computed within the test, so the test is comparing `f(x) == f(x)` where `f` is defined inline. If the component's actual logic changed, these tests would still pass. Specifically:
  - `shouldRenderText()` checks only `zoom >= 0.5` but the component (`ExplorerCanvas.svelte:3282-3283`) also requires `sw > 30 && sh > 14`.
  - `isVisibleAtZoom()` omits the query-matched override (`ExplorerCanvas.svelte:2505-2507`).
  - The viewport culling `isVisible()` and edge culling `isEdgeVisible()` are mathematically equivalent to the component, but still test their own copies rather than the actual rendering path.
  
  **Fix:** These tests should exercise the component itself — render ExplorerCanvas with specific zoom levels and node counts, then assert on observable effects (draw call counts, rendered elements). If the unit-test approach is intentional (testing the algorithm in isolation), the local functions must be extracted from the component and imported by both the component and the tests — not duplicated.

- [ ] **F3: Assertionless responsive layout tests — 4 of 5 ExplorerView-responsive tests have tautological assertions.** The responsive layout tests (`ExplorerView-responsive.test.js`) claim to verify chat collapse behavior, CSS breakpoints, and DOM structure, but 4 of 5 test functions assert only `expect(container).toBeTruthy()` — which always passes as long as `render()` returns anything. Specific issues:
  - `'has chat collapse button in template'` (line 90): Queries `.chat-collapse-btn` and `.chat-expand-btn` but never asserts on them — asserts on `.explorer-view || .ws-repo-list` instead.
  - `'chat expand button toggles chat visibility'` (line 102): Comments describe what SHOULD be tested but ends with `expect(container).toBeTruthy()` — no click interaction, no visibility toggle assertion. The test name claims toggle behavior is verified; the body does not verify it.
  - `'renders cleanly regardless of viewport width'` (line 116): Asserts `hasValidRoot` is truthy — identical to the first test.
  - `'chat area has collapse bar with button'` (line 127): Asserts only `expect(container).toBeTruthy()`.
  
  These tests satisfy the acceptance criterion "`npm test` passes" trivially, but provide zero functional coverage of the responsive layout behavior they claim to test. The chat collapse/expand toggle — the core interactive behavior added by this task — is completely untested. **Fix:** Test the actual collapse/expand interaction: render the component, click `.chat-collapse-btn`, assert `.explorer-chat-area` is removed from DOM and `.chat-expand-btn` appears, then click `.chat-expand-btn` and assert the chat area returns.

- [ ] **F4: Chat panel unrecoverable on viewport resize from medium to wide.** When `chatCollapsed` is set to `true` on a medium viewport (768px–1024px) via the collapse button, the chat area is removed from the DOM (`{#if !chatCollapsed}` — Svelte conditional rendering, line 1789). If the user then resizes to a wide viewport (>1024px), the CSS hides the expand button (`display: none !important` in `@media (min-width: 1025px)`, line 3646). The chat area remains gone with no way to re-open it — the collapse button is removed from DOM (chat is gone), and the expand button is CSS-hidden. The `chatCollapsed` state persists across viewport changes but the UI controls to toggle it are hidden. The task's acceptance criterion "Chat panel collapsible on medium viewports" implies the collapse is a medium-viewport-specific behavior, but the implementation makes it a global state change that can strand the UI. **Fix:** Either reset `chatCollapsed = false` when viewport width exceeds 1024px (via a `matchMedia` listener or `$effect`), or use CSS-only visibility (`display: none` on `.explorer-chat-area`) instead of Svelte conditional rendering for the collapse behavior, so the chat area always exists in the DOM and CSS media queries can control it.

- [ ] **F5: Performance benchmark test with 10k nodes has weak assertions that cannot verify the acceptance criterion.** The acceptance criterion states: "Explorer canvas renders 10k+ node graphs without dropping below 30fps during pan/zoom." The test `'renders a graph with 10k+ nodes without throwing'` (line 163) only asserts `not.toThrow()`. The test `'viewport culling reduces draw calls for off-screen nodes'` (line 203) asserts `fillRect calls < 50000` — an arbitrary ceiling that doesn't correlate to frame rate. No test measures render time or frame rate. The `'10k graph shows correct node count in stats'` test (line 184) uses `toMatch(/\d{4,}/)` which matches any 4+ digit number — it would pass for 1000 nodes or 99999 nodes. The implementation plan requires "performance test that loads a mock graph with 10k nodes and 20k edges, measures initial render time and pan/zoom frame rate. Target: <100ms initial render, >30fps during interaction." None of these targets are asserted. **Fix:** Add timing assertions: measure `performance.now()` before and after render and assert `< 100ms`. For pan/zoom frame rate, a jsdom test cannot measure real fps, so document this limitation and add a manual test instruction — but the render-time assertion is feasible and should be present.
