## Findings

### R1

- [-] [process-revision-complete] **F1 — DerivedInput signed by the new agent's key instead of the orchestrator's (spawner's) key — signing entity mismatch (spec §4.1, §4.5)**
  `crates/gyre-server/src/api/spawn.rs` lines 1465–1535 (`create_derived_input_for_agent`): The function generates a new Ed25519 keypair for the agent being spawned and uses that key to sign the `DerivedInput`. It then attaches the agent's `KeyBinding` (with `user_identity: "agent:{agent_id}"` — the new child agent's ID) to the `DerivedInput`. The spec §4.1 defines `DerivedInput.signature` as "orchestrator signs derivation" and `DerivedInput.key_binding` as "orchestrator's workload key binding." The spec §4.5 states: "Orchestrators sign derived inputs with workload-bound key." The orchestrator is the spawner (`auth.agent_id` / `spawner_id` parameter), not the agent being spawned. The current implementation means the DerivedInput proves the new agent authorized its own spawn — which is circular and meaningless for delegation provenance. The whole point of DerivedInput is to prove that a specific orchestrator (parent) authorized the delegation. Downstream chain verification (§4.4 step 1) will verify the DerivedInput signature against the key binding's public key, but since both belong to the child agent rather than the orchestrator, the verification proves nothing about who authorized the spawn. The correct implementation should: (a) retrieve the orchestrator's (spawner's) existing workload key from storage (it was stored when the orchestrator itself was spawned), (b) use the orchestrator's key to sign the derivation, (c) attach the orchestrator's `KeyBinding` to the `DerivedInput`, (d) separately generate and store the agent's keypair for the agent's own future use (signing output attestations at push time). **Fix:** Look up the spawner's signing key from `kv_get("agent_signing_keys", spawner_id)` and key binding from `kv_get("agent_derived_inputs", spawner_id)` (or the chain attestation for the spawner's task). Use the spawner's key to sign the DerivedInput and attach the spawner's KeyBinding. Generate the new agent's keypair separately and store it for the agent's push-time use.

- [-] [resolved] **F2 — Merge-time enforcement notification priority is 3 but spec requires 2 — task-vs-spec transcription error (repeat of TASK-007 R1 F2)**
  `crates/gyre-server/src/constraint_check.rs` line 678: `create_violation_notifications_with_priority(state, ..., 3)` assigns priority 3 to merge-time constraint violation notifications. The code comment (line 672) says "Phase 3: priority-3 notifications for merge-time violations (§7.5)" and the function doc comment (line 1399) says "Phase 3 merge-time uses priority 3 (§7.5: 'the push/merge was already rejected')." Both cite §7.5 but use the wrong number. The spec §7.5 (authorization-provenance.md line 754) states: "The human operator's inbox receives a notification (priority 2 — high but not critical, since the push/merge was already rejected)." The spec makes no distinction between push-time and merge-time priority — all constraint violation notifications are priority 2. The task file (task-008.md line 29) erroneously transcribed this as "priority-3 notification." This is the exact same finding class as TASK-007 R1 F2, which was already fixed for the `NotificationType::ConstraintViolation` default. The Phase 3 code re-introduces the wrong priority by explicitly overriding the correct default via `create_violation_notifications_with_priority`. Note: push-time enforcement correctly uses `create_violation_notifications` (line 443) which inherits the correct default priority 2 — only merge-time enforcement has this bug. **Fix:** Change `3` to `2` on line 678, or use `create_violation_notifications` (which already defaults to priority 2) instead of the `_with_priority` variant.

- [-] [partially-resolved → see R2 F5] **F3 — Verification endpoint returns chain-structure checks only, not the full VerificationResult tree with constraint evaluation (spec §6.2, §6.4)**
  `crates/gyre-server/src/api/provenance.rs` lines 143–174 (`get_verification`): The handler calls `verify_attestation_audit_only(&attestation)` (line 167) which performs structural verification only: checks non-empty `spec_path`/`spec_sha`, Ed25519 signature verification, key binding expiry, and `valid_until`. This covers only phase 1 of the §6.2 verification algorithm. The spec §6.2 defines a 5-phase algorithm: (1) verify input chain, (2) collect all constraints (explicit + strategy-implied + gate), (3) build CEL context from actual output, (4) evaluate all constraints, (5) verify output signatures. The endpoint is documented (both in its own doc comment and in api-reference.md) as returning the "full VerificationResult tree" — but it omits phases 2–5. A consumer calling this endpoint to verify a commit's attestation gets a `valid: true` result even if the commit violates constraints, because constraints are never evaluated. The `enforce_push_constraints` function (in the same file, `constraint_check.rs`) demonstrates the complete evaluation pipeline — it collects constraints, builds CEL context, evaluates, and reports. The verification endpoint should use a similar pipeline. **Fix:** Extend `get_verification` to: (a) load the full attestation chain (not just the single attestation), (b) find the root `SignedInput`, (c) derive strategy-implied constraints, (d) collect gate constraints, (e) compute the diff for the commit, (f) build a CEL evaluation context, (g) evaluate all constraints, and (h) return the full tree including per-constraint pass/fail results. The enforcement functions already implement this logic — extract and reuse it.

- [-] [resolved] **F4 — Export endpoint returns empty `trust_anchors` array, making offline verification impossible (spec §6.3)**
  `crates/gyre-server/src/api/provenance.rs` line 232: The `VerificationBundle` is constructed with `"trust_anchors": []` and a comment "trust anchors would be loaded from tenant config." The spec §6.3 defines `VerificationBundle.trust_anchors` as `TrustAnchor[] — public keys / JWKS` and states the bundle "can be verified by external compliance tools, auditors, federated Gyre instances." Without trust anchors, an offline verifier cannot verify any signatures in the chain — Ed25519 signature verification requires the signer's public key, and trust anchors are needed to validate key bindings against their issuers. The `state.trust_anchors` repository exists (`AppState.trust_anchors: Arc<dyn TrustAnchorRepository>`, wired in `lib.rs` line 281) and is populated via the `POST /api/v1/trust-anchors` CRUD API (trust_anchors.rs). The endpoint has access to the `state` and the attestation chain (which contains `trust_anchor_id` references in each `KeyBinding`). **Fix:** Load the trust anchors referenced by the attestation chain's key bindings: (a) collect all unique `trust_anchor_id` values from key bindings in the chain, (b) load each via `state.trust_anchors.find_by_id(...)`, (c) include them in the bundle's `trust_anchors` array. If a referenced trust anchor is not found, include a placeholder with the ID and a note that it was unavailable.
  **Resolution (TASK-009 commit 55c1fc20):** Trust anchors are now loaded from the tenant via `state.trust_anchors.list_by_tenant()` using the workspace's tenant_id. Verified at `provenance.rs:243-258`.

### R2

- [-] [process-revision-complete] **F1 (PERSISTS from R1) — DerivedInput still signed by child agent's key instead of orchestrator's (spawner's) key — signing entity mismatch (spec §4.1, §4.5)**
  `crates/gyre-server/src/api/spawn.rs` line 1517: The code comment says "Sign the derivation with the agent's ephemeral key." The keypair generated at line 1478 is used for both the KeyBinding (`user_identity: "agent:{agent_id}"` — the child agent, line 1499) and the DerivedInput signature (line 1528). The spec §4.1 requires `DerivedInput.signature` = "orchestrator signs the derivation" and `DerivedInput.key_binding` = "orchestrator's workload key binding." The spec §4.5 states "Orchestrators sign derived inputs with workload-bound key." The orchestrator is the spawner (`spawner_id` parameter), not the agent being spawned. This finding was identified in R1 F1 and confirmed by `scripts/check-crypto-verify.sh` Class 3 (still triggers). The R1 fix was a process revision (adding the Class 3 check) — the code was not changed. The correct implementation should: (a) load the spawner's existing workload key from `kv_get("agent_signing_keys", spawner_id)`, (b) load the spawner's KeyBinding from the spawner's attestation chain, (c) use the spawner's key to sign the DerivedInput and attach the spawner's KeyBinding, (d) separately generate and store the child agent's keypair for future push-time use (this part already works correctly). **Impact:** DerivedInput proves the child authorized its own spawn, which is circular. Downstream chain verification (§4.4 step 1) verifies the DerivedInput signature against the key binding's public key, but since both belong to the child agent, the verification proves nothing about who authorized the delegation. The security property "a compromised agent cannot forge delegations from its parent" is not enforced.

- [-] [process-revision-complete] **F5 — Verification endpoint still performs only structural verification (Phase 1), not the full §6.2 algorithm (spec §6.2, §6.4)**
  `crates/gyre-server/src/api/provenance.rs` lines 167-186 (`get_verification`): The handler calls `verify_chain(&chain, max_depth)` which performs only Phase 1 of the §6.2 verification algorithm — structural checks (Ed25519 signature verification for SignedInputs, key binding expiry, valid_until, chain depth, parent linkage, root SignedInput presence). Phases 2–5 are not performed: (2) collect all constraints (explicit + strategy-implied + gate), (3) build CEL context from actual output, (4) evaluate all constraints, (5) verify output signatures. The R1 F3 fix improved the endpoint from calling `verify_attestation_audit_only` (single-node) to `verify_chain` (multi-node structural), but the core issue remains: the endpoint returns `valid: true` when structural checks pass, even if the commit violates output constraints. The acceptance criterion says "Verification endpoint returns complete VerificationResult tree." The enforcement code in `constraint_check.rs` (`enforce_push_constraints`, `enforce_merge_constraints`) already implements Phases 2–5 — constraint collection, CEL context building, and evaluation. The verification endpoint should use the same pipeline. **Fix:** Extend `get_verification` to: (a) find the root SignedInput from the chain, (b) derive strategy-implied constraints, (c) collect gate constraints via `accumulate_chain_constraints`, (d) compute the diff for the commit, (e) build a CEL evaluation context, (f) evaluate all constraints, (g) check output signatures if present, and (h) include per-constraint pass/fail results in the VerificationResult tree. The existing enforcement functions demonstrate the complete pipeline.
