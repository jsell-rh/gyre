# Database & Migration Strategy

## ORM: Diesel

**Diesel** is Gyre's ORM and migration framework. It provides:
- Compile-time schema validation via generated `schema.rs`
- Type-safe query builder (no raw SQL in application code)
- Native support for both SQLite and Postgres from the same codebase
- Built-in migration CLI (`diesel migration generate/run/revert`)
- Paired up/down migrations by default

### Why Diesel

- Most popular Rust ORM - large ecosystem, well-documented, battle-tested
- Compile-time guarantees catch schema drift before runtime
- Dual-backend support aligns with Gyre's abstracted storage principle (SQLite default, Postgres for enterprise)
- Migration tooling is built in, not bolted on

## Migration Framework

### File Structure

```
crates/gyre-adapters/
  migrations/
    00000000000000_diesel_initial_setup/
      up.sql
      down.sql
    2026-03-20-000001_create_core_tables/
      up.sql
      down.sql
    2026-03-20-000002_create_review_tables/
      up.sql
      down.sql
    ...
  src/
    schema.rs          # Auto-generated by Diesel CLI
    sqlite/
      mod.rs           # SQLite-specific adapter (if needed)
    postgres/
      mod.rs           # Postgres-specific adapter (if needed)
```

### Migration Rules

1. **Every migration has both `up.sql` and `down.sql`.** No exceptions.
2. **Destructive downs must be flagged.** If a `down.sql` drops a column or table that loses data, the migration file must contain a comment: `-- DESTRUCTIVE: drops column X, data loss on rollback`. BCP snapshot should be taken before applying.
3. **Migrations are transactional.** Each migration runs inside a transaction. If any statement fails, the entire migration rolls back.
4. **Migrations are idempotent where possible.** Use `CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`, etc.
5. **No application logic in migrations.** Migrations are pure DDL/DML SQL. No Rust code, no API calls, no external dependencies.
6. **Both backends tested.** Every migration must work on both SQLite and Postgres. If a migration requires backend-specific SQL, provide both variants.

### Migration Lifecycle

```
diesel migration generate <name>    # Create up.sql + down.sql stubs
diesel migration run                # Apply pending migrations
diesel migration revert             # Roll back last migration
diesel migration redo               # Revert + run (test roundtrip)
diesel print-schema                 # Regenerate schema.rs
```

## Startup Behavior

### Auto-Migrate on Start

When the Gyre server starts:

1. **Acquire migration lock.**
   - Postgres: advisory lock (`SELECT pg_advisory_lock(...)`)
   - SQLite: file lock on the database
   - Only one process migrates at a time.

2. **Check schema version.**
   - Read `__diesel_schema_migrations` table.
   - Compare against embedded migration list.

3. **Evaluate state:**
   - DB behind binary (pending migrations) -> **auto-migrate**, then start.
   - DB matches binary -> **start normally**.
   - DB ahead of binary (binary is older than schema) -> **refuse to start** with error: `"Database schema version X is ahead of binary version Y. Upgrade the binary or roll back the database."`

4. **Release migration lock.**

5. **Start serving requests.**

### Crash-Backoff for Locks

In Kubernetes with multiple pods:
- First pod acquires the migration lock and runs migrations.
- Other pods fail to acquire the lock, log a warning, and crash.
- Kubernetes restart backoff handles retry naturally.
- Once migrations complete, subsequent pod starts succeed.

This is a deliberate design choice - simpler than distributed coordination, and Kubernetes handles it for free.

## Multi-Tenancy: Row-Level Isolation

### Approach

Shared schema with `tenant_id` column on all tenant-scoped tables. Diesel's query builder enforces tenant filtering at the application layer.

### Which Tables Get tenant_id

**Tenant-scoped (have tenant_id):**
- projects
- repositories
- agents
- tasks
- merge_requests
- review_comments
- reviews
- merge_queue
- agent_commits
- agent_worktrees
- activity_events
- analytics_events
- cost_entries
- audit_events
- quality_gates
- gate_results

**Global (no tenant_id):**
- users (a user can belong to multiple tenants)
- api_keys (scoped to user, not tenant)
- siem_targets (may be global or per-tenant - TBD)
- network_peers (infrastructure-level)
- `__diesel_schema_migrations` (system table)

### Tenant Enforcement

```rust
// Every query on a tenant-scoped table MUST include tenant filtering.
// Diesel's type system helps here - the tenant_id column is non-optional.

fn list_projects(conn: &mut DbConn, tenant: &TenantId) -> Vec<Project> {
    projects::table
        .filter(projects::tenant_id.eq(tenant))
        .load(conn)
}
```

**Enforcement mechanisms:**
- Compile-time: Diesel query builder makes it hard to forget (column is NOT NULL)
- Runtime: Middleware extracts tenant from auth context and injects into all queries
- Test: CI includes tests that verify no tenant-scoped query runs without a tenant filter
- Lint: Custom lint or architecture test that greps for raw table access without `.filter(tenant_id.eq(...))`

### Tenant Resolution

Tenant is resolved from the authenticated principal:
- JWT claims include tenant_id (from Keycloak realm/org)
- API key resolves to user -> user's active tenant
- Agent token resolves to agent -> agent's tenant
- Global auth token -> system tenant (admin access to all)

## Testing Strategy

### Migration Tests (CI)

1. **Full chain test:** Apply all migrations from empty on both SQLite and Postgres. Verify final schema matches `schema.rs`.
2. **Roundtrip test:** For each migration, run `up` then `down` then `up` again. Verify no errors.
3. **Data preservation test:** For migrations that transform data, seed test data before migration, verify it's correct after.
4. **Downgrade protection test:** Verify server refuses to start when DB schema is ahead of binary.
5. **Lock contention test:** Verify second process waits/crashes when first holds migration lock.

### Backend Parity Tests

Every query in the adapter layer runs against both SQLite and Postgres in CI. Diesel's abstraction handles most differences, but edge cases (JSON handling, date functions, text collation) must be tested on both.

### Test Infrastructure

```
# CI matrix:
- SQLite (in-memory for speed, file-backed for migration tests)
- Postgres (via testcontainers-rs or CI service container)
```

## Migration from Current Implementation

The current hand-rolled rusqlite implementation (8 inline migrations, raw SQL queries throughout `gyre-adapters`) must be replaced. This is a significant refactor:

1. **Generate Diesel migrations** matching the current 8 inline migrations (preserving schema compatibility).
2. **Generate `schema.rs`** from the resulting schema.
3. **Rewrite adapter layer** to use Diesel query builder instead of raw rusqlite.
4. **Add Postgres support** via Diesel's backend abstraction.
5. **Add `tenant_id`** to all tenant-scoped tables (migration + query updates).
6. **Add migration lock** for concurrent startup safety.
7. **Add downgrade protection** check on startup.
8. **Update CI** to test both backends.

### Data Migration Path

For existing Gyre deployments (SQLite):
- Diesel's migration table (`__diesel_schema_migrations`) is separate from the current `_migrations` table.
- A one-time bootstrap migration reads the old `_migrations` table, marks equivalent Diesel migrations as applied, then drops `_migrations`.
- Existing data is preserved. No re-import needed.

## Relationship to Existing Specs

- **Architecture** (`development/architecture.md`): Storage is behind a port. Diesel lives in `gyre-adapters`, never in `gyre-domain`.
- **Business Continuity** (`system/business-continuity.md`): BCP snapshots should be taken before destructive migrations. Restore must account for schema version.
- **Supply Chain** (`system/supply-chain.md`): Schema version is part of the deployment artifact. AIBOM should include DB schema version.
- **Design Principles** (`system/design-principles.md`): "No shortcuts" - Diesel's compile-time checks enforce this at the data layer.
